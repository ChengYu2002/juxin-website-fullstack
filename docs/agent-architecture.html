Jason 是怎么"查着库说话"的

一个用 LangGraph 编排的工具调用 Agent：模型不再凭空答产品问题，而是先查真实数据库、只复述查到的真值。这页拆开架构、文件、和面试时该讲的点。

编排 LangGraph.js 模型 qwen-plus + 寒暄 qwen-flash 端点 DashScope OpenAI 兼容 后端 Express + Mongoose 前端 React + react-markdown

一句话总览

用户在聊天气泡里问产品（"推荐两款手推车"、"JX-80SP 装箱量多少"），后端把问题交给一个 Agent 循环：模型判断要不要查库 → 调工具查真实产品数据 → 拿到结果后再组织回答。核心目标是杜绝幻觉——型号、规格、链接全部来自数据库，模型只做"翻译和组织"，不许自己编。

价格 / 库存 / 交期这类库里没有的，一律礼貌转人工并引导留资。

01

请求全链路

一条消息从前端到模型再回来，依次经过这些层。左侧色条标层：■ 前端 · ■ Agent · ■ 工具/数据。

前端 · 输入
ChatWidget.jsx → services/chatService.js

用户输入，前端持有整段对话历史，POST /api/chat 把 messages[] 整包发过去（模型无状态，历史即记忆）。

↓
路由 + 控制器
routes/chat.js → controllers/chatController.js

限速；校验 messages；后端注入 system prompt（不信任前端传的身份，防 prompt 注入）；把 {role,content} 转成 LangChain 消息对象，调 runAgent()。

↓
编排 · LangGraph 图
agent/graph.js

StateGraph：START → agent 节点。agent 节点 = ChatOpenAI(qwen-plus).bindTools()，模型决定是否调工具。

Agent 循环（ReAct）：agent 节点输出里带 tool_calls → 条件边走到 tools 节点（ToolNode 执行工具、把结果作为 ToolMessage 喂回）→ 回到 agent 节点再判断。没有 tool_calls 时 → END，输出最终文本。
↓
工具层
agent/tools.js — searchProducts / getProduct

用 tool() + zod schema 定义。搜索工具只返回"识别用"字段（型号/名称/链接/是否热销）；取详情工具返回瘦身后的完整规格。

↓
数据服务层
services/productService.js → models/product.js (MongoDB)

listPublic() / getPublicByIdOrSlug() 直接查库。工具直接调 service，不绕自己的 HTTP 接口——少一次网络、复用同一套查询逻辑。

↑ 结果喂回模型，模型组织成最终回答
前端 · 渲染
ChatWidget.jsx (react-markdown)

回复用 Markdown 渲染，产品链接 [JX-80SP](/products/jx-80sp) 变成可点蓝链；点站内链接走 SPA 跳转并收起面板。

03

核心概念（面试主线）

① LangGraph StateGraph 与 Agent 循环

图有两个节点：agent（调模型）和 tools（执行工具）。条件边看模型最新输出里有没有 tool_calls——有就去 tools，执行完喂回再回 agent；没有就 END。这就是 ReAct 循环：想→调工具→看结果→再想，直到能回答。状态用 MessagesAnnotation 累积消息。

START → agent → 有 tool_calls? ─是→ tools → 回 agent
                        └─否→ END

② Tool Calling（函数调用）

工具用 tool(fn, {name, description, schema}) 定义，schema 是 zod。模型不直接执行工具——它输出一段结构化的 tool_calls（工具名 + 参数 JSON），由 ToolNode 执行、把返回值作为 ToolMessage 塞回对话，模型下一轮据此作答。description 写得好不好直接影响模型调不调、调得准不准。

③ 防幻觉：防御纵深（四道）

单一措施都堵不死幻觉，用多道叠加：

  • ① 强制工具（tool_choice: 'required'）：命中产品意图时逼模型必须调工具——堵"跳过工具凭感觉编型号"。
  • ② 瘦身工具返回：搜索列表只给型号/链接/是否热销、不给数字，避免多条间把 MOQ/颜色数张冠李戴；具体数值只从单条 getProduct 取。
  • ③ Prompt 收紧 + temperature 0：只准说工具返回的字段，功能/卖点不许编。
  • ④ 出口校验兜底（最硬）：回复发出前把每个 /products/<id> 链接拿去库里核——有假的→强制调工具重试一次→仍假则返回安全兜底话术。假链接绝对到不了用户，不再依赖"永远堵不全"的关键词正则。

④ 领域知识 + 多维检索

品类词典写进 system prompt（业务知识，非幻觉）：购物手推车=带袋·买菜；多用途手推车=无袋·承重·搬家。模型据此听懂需求、选对 category、跟客户解释区别。检索维度：searchProducts 支持 分类 + 关键词 + 颜色（中文颜色由模型翻成英文再传，库里对颜色名做不区分大小写子串匹配）。结构化特征：getProduct 返回 features(foldable/brake/伸缩…) + loadCapacity(承重) + material(材料)，助理据此做"搬重物→高承重金属材料""要收纳→可折叠"的有据推荐——但仍只说工具真返回的值，没返回就当没有。

⑤ Provider 抽象

ChatOpenAI 通过 configuration.baseURL 指向 DashScope 的 OpenAI 兼容端点。换模型/换厂商只改 3 个 env（LLM_BASE_URL / LLM_API_KEY / LLM_MODEL），代码零改。部署位置不同选不同 provider（墙外可 Gemini、墙内用 DashScope/DeepSeek）。

⑥ 无状态 + 分层

模型无记忆：前端持有对话历史，每次整包发，后端不落库（落库是 P4）。分层清晰：route → controller → agent(graph) → tools → service → model，工具直接调 service 不绕 HTTP。system prompt 后端注入，前端只发消息内容，防 prompt 注入。

⑦ 模型分级（速度 ↔ 可靠）

按任务难度用不同模型：纯寒暄/问候（锚定白名单命中）走 qwen-flash 快车道（无工具无图，~0.9s 秒回，没产品数据可编所以安全）；产品问答/推荐才走 qwen-plus + 工具主线（可靠但 2 轮 ~4–6s）。这是刻意的速度↔可靠取舍：flash 快但会跳工具/编型号，plus 慢但听话。

⑧ 语言一致

system prompt 整体是中文，短英文问句容易被带偏成中文回复、或中英混排。解法：每轮按买家最近消息的语言，在消息末尾动态追加一条强指令（中文→"全程中文别夹英文"；非中文→"reply entirely in the buyer's language, no Chinese"）。放末尾 + 用目标语言写，压得住。

04

值得注意的坑

!

弱模型不肯调工具 → 编假型号 → 死链

qwen-flash 对"推荐"这类软请求会跳过工具、凭命名规律脑补出 JX-180S 这种库里不存在的型号（链接 404）。qwen-plus + 强制 tool_choice 才可靠。"为什么会推荐不存在的产品" = LLM 本质是续写不是查库。

!

关键词正则强制"永远堵不全"

用正则判断"是不是产品意图"来决定要不要强制调工具，换个说法就绕过（"家里搬东西用啥好"不含关键词→没强制→模型编 JX-UT800 还自称"真实在售"）。所以不能只靠正则——必须加出口校验兜底（第③节第 4 道）兜住漏网的。

!

tool_choice: 'required' 只能用在"第一轮"

若每轮都强制调工具会死循环（拿到结果还被逼再调）。用 hasToolResult（消息里是否已有 ToolMessage）判断，只在首轮强制，之后放开让它作答。

!

产品的 name 和 url-id 可能不一样

如 id=jx-f4-2 但 name=JX-F4。链接用 id，显示名用 name，两者都以工具返回为准，别自己拼。

!

LLM 没有 100% 保证 → 已加出口校验兜底

强制工具让它极可靠，但理论上仍可能无视结果乱写。所以发出前把回复里每个 /products/xxx 链接对库校验（见第③节第 4 道，已实现于 graph.js 的 runAgent）：有假链接→强制调工具重试一次→仍假则返回安全兜底话术，假链接绝不外泄。

!

LangGraph 的 ChatOpenAI 有自己的重试

P1 那套自定义 429 退避（llm.js）在图里不生效，改用 ChatOpenAI 的 maxRetries；日配额耗尽的 429 由 controller 兜底成友好提示。

05

面试问答演练

点开看参考答法。用自己的话讲，别背。

Q讲讲这个 Agent 的整体架构？
前端浮动气泡持有对话历史，整包发到 /api/chat。后端注入 system prompt 后交给一个 LangGraph StateGraph：agent 节点调模型（ChatOpenAI 指向 Qwen），模型决定是否调工具；条件边根据 tool_calls 在 agent 和 tools 节点间循环。工具（searchProducts / getProduct）直接调 productService 查 MongoDB，返回瘦身数据喂回模型组织成答案。分层：route → controller → graph → tools → service → model。
Q怎么防止 AI 编造/幻觉？
防御纵深，四道叠加（单一措施都堵不死）：①产品意图强制调工具（tool_choice=required）；②工具返回瘦身，搜索列表只给型号/链接不给数字，避免多条串味；③prompt"只准说工具字段"+ temperature 0；④出口链接对库校验——发出前核每个产品链接，假的就强制重试、再假则安全兜底，假链接绝不外泄。再叠加换用遵循度更高的 qwen-plus。关键洞察：正则判意图会漏，必须有出口兜底。
Q为什么用 LangGraph，而不是自己写个 while 循环？
循环本身简单，但 LangGraph 给的是可扩展的结构：状态管理（MessagesAnnotation）、标准化的 ToolNode、条件边把控制流显式化。后面加记忆（P4）、加 RAG 检索节点（P5）、加人工审核断点，都是在图上加节点/边，不用重写循环。也更好观测和调试。
Q工具是怎么定义的？为什么直接调 service 不调自己的 API？
用 tool() + zod schema 声明入参，函数体调 productService。直接调 service：省一次 HTTP 往返、少一层出错面、和 controller 复用同一套查询逻辑（单一数据源）。为此专门把查询从 controller 抽成了 productService。
Q换模型 / 换厂商麻烦吗？
不麻烦。走 OpenAI 兼容协议，ChatOpenAI 用 baseURL 指到 DashScope；换厂商只改 LLM_BASE_URL / LLM_API_KEY / LLM_MODEL 三个环境变量，代码不动。还能按部署位置选（墙内墙外）。
Q对话历史 / 记忆怎么处理？
现阶段模型无状态：前端持有 messages 历史，每轮整包发过来，后端只负责当轮推理不落库。好处是简单、可横向扩展。持久化对话 + 关联询盘是下一阶段（P4）的事。
Q响应为什么慢？还能优化吗？
产品问答 = 2 次模型往返（决定调工具 → 组织答案），每次 qwen-plus 约 2–3s，所以 ~4–6s 是"可靠模型 + 工具 + 发出前校验"三条不变时的结构性地板，安全范围内砍不动。已做的安全优化：简洁回复、maxTokens 封顶、寒暄走 flash（~0.9s）。再往下只剩 流式（总时长不变，但 2–3s 就开始出字、体感减半；寒暄类没 guardrail 冲突可安全流式）。另外免费/低配额档会限流——压测后单轮飙到 10s+ 是限流假象，非代码问题，生产付费档 + 低流量不会。
Q防幻觉这套是不是补丁太多？如果重来会怎么设计？
诚实说：现在是反应式打地鼠（强制正则 + 瘦身 + prompt + 出口校验 + 兜底），防御纵深有效但不优雅——是在补"模型爱编 + 不够听话"。更优雅的架构："LLM 只选、代码渲染真值"——工具返回带 id 的数据 → LLM 只输出结构化决策（如 {recommend:[id…]}）→ 后端按 id 从库取真实名/链接/规格，由代码渲染（或产品卡片）。这样 LLM 结构上没机会吐出假产品，能替掉大半补丁。代价是少点自由行文，但产品卡 UX 可能更好。能讲出这个 = 显架构成熟度。
Q成本 / 更换更便宜快的模型？
聊天流量低，qwen-plus 已经很便宜（10 元用很久）。要更快更省可 A/B：deepseek-v4-flash（便宜、支持 tool calling、自带 prompt 缓存正好治"大 system prompt 每轮发两遍"）、qwen-turbo（零迁移）、GLM-4-Flash。但换模型必须先跑 eval 防幻觉倒退——"支持工具调用" ≠ "可靠不编"。这也是先建 eval 的价值：它是换模型的安全网。
Q怎么保证改架构不把行为改坏？
一套行为回归 eval（scripts/agent-eval.js，35 条）：打 /api/chat + 确定性断言 + 出分。关键——真值从数据库现取自推（期望值直接读库），不靠硬编、不会因记错某个数误报；断言只认真值不认措辞、数字去逗号归一。改前跑拿基线，改后必须 ≥ 基线，红绿一眼见。把"打地鼠"变"照表迭代"。（调 eval 的坑：颜色标签会本地化 Black→黑色，不能比英文原文。）
Q用户问政治 / 医疗 / 无关问题怎么办？
检测 → 固定挡回，绝不让模型自由生成。政治/地缘(台湾)/宗教/医疗/其他公司/编程等一律不表态、不展开，礼貌转回产品；情绪(分手)先一句共情再温和转回。要点：面向全球买家的销售 bot 就地缘政治表态本身就是风险（无论说什么），所以检测到就挡回，而非让模型去"答"。当前 prompt 规则版；更彻底是 flash 路由把 off-topic 分敏感/情绪/普通三档。
