01背景与目标

巨鑫是一家外贸制造企业(购物手推车、多用途手推车、露营拖车、户外家具),核心业务链路是:访客提交询盘 → 邮件通知业务员 → 人工跟进。海外买家的常见问题高度结构化 —— MOQ、装箱量、集装箱装载、包装、能否改颜色/OEM —— 而这些答案恰好已存在产品库(specs.containerLoad、moq、variants…)。

目标:用一个售前 AI 助理把业务员重复回答的问题前置到官网自动答掉,并把高意向访客转化成一条更完整的结构化询盘,同时对超出能力边界的问题(报价/库存/交期)干净地转人工。

  • 业务目标 —— 降低业务员重复答疑负担;提升询盘质量与转化。
  • 工程目标 —— 在真实业务系统中落地一个有状态、有工具调用、有 RAG 的 Agent。

02定位与非目标

这是全文最重要的一节 —— 边界定义了"不像玩具"与"过度承诺"之间的线。

定位:售前助理(pre-sales),不是成交客服。 它帮买家问清产品、做选型、并留资,不负责报价与交易。

06Agent 状态设计

分清三种"状态":UI 运行状态(前端 spinner)、对话数据状态(要 carry)、控制流状态(LangGraph 节点)。

① 数据状态(跨轮次携带)

// conversations 集合里一条会话的状态
{
  conversationId,          // 匿名号牌(localStorage)
  messages: [...],         // 历史,喂回模型 = "记忆"
  collectedLead: {         // 边聊边攒,决定能否写询盘
    name?, email?, product?, quantity?, destination?
  }
}

② 控制流状态(LangGraph 状态图)

图 2 · LangGraph 状态图。只建"结构性"状态(有无 lead / 是否越界 / 是否在工具调用),对话措辞交给 LLM。

避坑

"查询中 / 回答中"不是 Agent 状态,是前端根据事件显示的 UI status,别建模进逻辑。状态别过度细分,否则 Agent 会僵、丢掉 LLM 的灵活性。

工具输入输出映射

硬约束

08RAG 设计

硬规格走 tool calling(精确);软信息(公司简介、认证、OEM/ODM、包装出货、FAQ)走 RAG。内容量小,pipeline 要真、规模可小。

  • 第 0 步 —— 先把软内容写出来(现状 company.mock.js 太薄),这是真正的工作量。
  • Pipeline —— chunk → embedding → 存 Atlas Vector Search → 相似度检索 → 注入 prompt。
  • 评估 —— 20 条软问题做 before/after,拿到"命中率从 X 提到 Y",对上 JD 的"RAG 准确率提升经验"。
决策

功能上内容少、静态块就够;但为 RAG + 向量库 关键词,升级成真 RAG(Atlas Vector Search / FAISS)。定位为"小而真"的 pipeline。

09模型选型

关键不是"智商",是 tool calling 可靠性 + 多语言 + 不幻觉。太小的模型(7B)在工具调用上会飘。甜点区是 Flash 级 / 70B 级 —— 而这一档有免费的。

方案免费Tool Calling可部署用途
Gemini Flash✔ 免费额度✔✔主力(托管)
Groq · Llama 70B✔ 免费额度✔✔主力 + "开源"标签
本地 Ollama✔ 永久⚠ 小模型弱✘ Render 无 GPU学原理 / 离线

用 OpenAI 兼容 SDK 写,endpoint 可换 —— "模型无关、可切换"本身是工程加分点。

10防幻觉与安全边界

  • 数字只用工具返回值 —— system prompt 明令"不得自行编造规格/价格/交期"。
  • 越界即转人工 —— 价格/库存/交期/下单一律引导提交询盘,给出统一话术。
  • 写询盘有守卫 —— createInquiry 需 name + email 才允许,防止误触发。
  • 输入清洗 —— 沿用现有 inquiry 校验;注意 prompt 注入,只信任工具/资料内容。

11会话与记忆

记忆 ≠ 账号。 模型无状态,"记忆"= 每次把历史一起发过去;关联只需一个匿名会话 ID,不需要登录。

  • 首次开聊,前端生成 UUID conversationId 存 localStorage,每次请求带上。
  • 后端按 conversationId 存 Mongo —— 无用户,一个 id 当钥匙。
  • 漏斗:浏览时匿名 → 留 email 提交询盘那一刻,把 conversationId 关联到 inquiry → 业务员看到完整售前对话背景。
  • 隐私:留资前不含 PII,合规干净。

13成功指标与评估

指标目标怎么测
规格问答准确率≥ 95%(数字类)20–30 条测试集,标 pass/fail
幻觉率(数字)≈ 0抽查是否只复述工具值
越界转人工命中≥ 90%价格/库存类问题是否正确拒绝
RAG 软问题命中before/after 对比eval 集报提升幅度

14开发里程碑 (P0 → P8)

从零学 Agent 的爬坡顺序。总时长:专注 ~1 周 / 下班式 ~2–3 周。

P0

练手沙盒(一次性)

独立小脚本,用免费 API 让模型调用一个假工具,亲眼看懂"模型→工具→喂回→回答"循环。

学:tool calling 消息格式 + 循环~0.5 天
P1

基础对话

Juxin 加 /api/chat(先不接工具)+ 最简聊天气泡。先让它会说话。

学:前后端聊天链路 · 流式1–2 天
P2

接工具 · 核心 ★

用 LangGraph 编排,把现有产品接口包成 tool,让它真的查库回答规格。

学:tool schema · LangGraph · 浅循环2–4 天
P3

防幻觉 + 护栏

写 system prompt:身份、数字只用工具值、越界转人工。

学:prompt 工程 · 边界控制1 天
P4

记忆 + 抓询盘 · 闭环 ★

conversationId 存 Mongo;create_inquiry 工具,信息够了落结构化询盘并关联对话。

学:把 Agent 接进真实业务动作1–2 天
P5

RAG + 向量库 ★

公司软信息切块 → Atlas Vector Search → 检索注入 + 小评估。(为 RAG/向量库关键词升级)

学:embedding · 检索 · 评估1–2 天
P6

严肃信号

20–30 条 eval 集(报准确率)+ 对话/工具日志接进 admin 后台。

学:评估思维 · 可观测性1–2 天
P7

部署

免费模型 key 配到 Render 环境变量,上线。

学:密钥管理 · 部署~0.5 天
P8

MCP(拉伸)

工具封装成 MCP server,agent 经协议消费。前沿加分,排最后。

学:MCP 协议 · 服务解耦1–2 天

15技术栈与部署

  • Agent —— Node + LangGraph.js(留在 Juxin 栈内);模型经 OpenAI 兼容 SDK,可切换。
  • 向量 —— MongoDB Atlas Vector Search(零新增基建)或 FAISS。
  • 封装 —— Docker 多阶段;若 agent 做独立服务,则 compose 编排 frontend + backend + agent 三容器(强化"多服务编排 + MCP"故事)。
  • 部署 —— Render 每服务单独部署;本地 compose 展示多服务能力即可,线上可简化。

16范围纪律(不做什么)

明确不碰 —— 超预算且易在面试翻车

多 Agent 编排 · 手写 ReAct 脚手架 / LangChain 老 AgentExecutor · 微调 / 自训 embedding · 复杂 RAG(re-rank / query 改写)· LangGraph 复杂状态机 · 正式 eval 框架。

定位是"全栈工程师把 AI 功能扎实落进真实系统",不是 Agent 研究。深度不是战场 —— 扎根 + 闭环 + 测量才是。ReAct 当概念理解即可,用原生 tool calling / LangGraph 实现。
